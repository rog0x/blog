MCP Server Security: Best Practices for Developers in 2026

March 25, 2026 · 7 min read · MCPSecurityDevelopment

MCP servers are powerful. They give AI models the ability to read files, execute commands, query databases, make HTTP requests, and interact with external services. That power is exactly why security matters. A poorly secured MCP server is not just a bug — it is a backdoor into your system that an AI model (or a malicious prompt) can exploit.

As the MCP ecosystem grows — there are now hundreds of community servers on npm and GitHub — the attack surface grows with it. This guide covers the security practices every MCP developer should follow, whether you are building servers for yourself, your team, or the public.

Why MCP Security Matters

Traditional software security focuses on human users. MCP security has an additional dimension: the AI model itself is the user. This creates unique risks:

  • Prompt injection — Malicious content in web pages, emails, or documents can instruct the AI to misuse MCP tools in ways the human user did not intend
  • Implicit trust — Users often trust AI suggestions without careful review, so a compromised tool call may go unnoticed
  • Broad access — MCP servers often have filesystem, network, or database access that far exceeds what any single tool invocation needs
  • Chain attacks — An attacker who compromises one MCP server can potentially use it as a pivot point to reach others in the same environment

The core principle is simple: an MCP server should do exactly what it claims to do, nothing more, and it should be impossible to make it do anything else through manipulated input.

Input Validation: Your First Line of Defense

Every tool in your MCP server accepts input. Every input is a potential attack vector. Validate everything, trust nothing.

Validate Types and Shapes

Use a schema validation library like Zod (TypeScript) or Pydantic (Python) to define exactly what your tool accepts. Reject anything that does not match.

import { z } from "zod";

// Define strict input schema
const FileReadInput = z.object({
  path: z.string()
    .min(1)
    .max(500)
    .refine(p => !p.includes(".."), "Path traversal not allowed")
    .refine(p => p.startsWith("/allowed/"), "Path must be in allowed directory"),
  encoding: z.enum(["utf-8", "ascii", "base64"]).default("utf-8"),
  maxBytes: z.number().int().min(1).max(10_000_000).default(1_000_000),
});

// In your tool handler:
const parsed = FileReadInput.safeParse(input);
if (!parsed.success) {
  return { error: `Invalid input: ${parsed.error.message}` };
}

Sanitize String Inputs

Strings are the most dangerous input type. They can contain path traversal sequences, shell metacharacters, SQL injection payloads, and prompt injection attempts. Apply these rules:

  • File paths — Resolve to absolute paths, then verify they are within allowed directories. Use path.resolve() to eliminate .. sequences, then check the prefix.
  • URLs — Parse with the URL constructor, validate the protocol (only https:), check against an allowlist of domains if possible.
  • Shell arguments — Never interpolate into shell commands. Use execFile with an argument array, not exec with a string.
  • SQL — Always use parameterized queries. Never concatenate user input into SQL strings.
  • Regex — If accepting regex patterns from users, set a timeout and complexity limit to prevent ReDoS attacks.
import path from "path";

function safePath(userPath: string, allowedRoot: string): string {
  // Resolve to absolute, eliminating .. and symlinks
  const resolved = path.resolve(allowedRoot, userPath);

  // Verify the resolved path is still within allowed root
  if (!resolved.startsWith(path.resolve(allowedRoot))) {
    throw new Error("Access denied: path outside allowed directory");
  }

  return resolved;
}

// Usage:
const filePath = safePath(input.path, "/home/user/projects");
// Even if input.path is "../../etc/passwd", this will reject it

Limit Input Size

Set maximum lengths on all string inputs. A tool that processes markdown does not need to accept a 500MB string. A URL parameter does not need to be 50,000 characters. Set reasonable limits and enforce them.

Authentication Patterns

Not every MCP server needs authentication — a local markdown formatter does not need API keys. But if your server accesses external services, handles sensitive data, or is exposed to a network, authentication is essential.

API Key Management

The most common pattern for MCP servers that connect to external APIs:

  • Environment variables — Read API keys from environment variables, never hardcode them
  • Validation on startup — Check that required keys are present when the server starts, fail immediately with a clear error if they are not
  • Key rotation — Document how to rotate keys and make it easy to do
  • Scope limitation — Use API keys with the minimum required permissions. If your server only reads data, do not use a key that can write.
// Good: Read from environment, validate on startup
const API_KEY = process.env.SERVICE_API_KEY;
if (!API_KEY) {
  console.error("SERVICE_API_KEY environment variable is required");
  process.exit(1);
}

// Bad: Hardcoded key
const API_KEY = "sk-1234567890abcdef"; // Never do this

OAuth for User-Facing Servers

If your MCP server acts on behalf of a user (accessing their GitHub, Google Drive, etc.), use OAuth 2.0 with PKCE. Store tokens securely, handle refresh token rotation, and always request the minimum scopes needed.

Server-to-Server Authentication

If your MCP server communicates with other internal services, use mutual TLS (mTLS) or signed JWTs with short expiration times. Do not rely on network-level security alone — assume the network is compromised.

Sandboxing: Limit the Blast Radius

Even with perfect input validation, defense in depth requires limiting what your server can do at the system level.

Filesystem Restrictions

An MCP server that analyzes JSON files should not have access to /etc/passwd or ~/.ssh/. Restrict filesystem access:

  • Allowlisted directories — Define exactly which directories the server can read from and write to
  • Read-only where possible — If a tool only needs to read files, do not give it write access
  • No access to sensitive paths — Explicitly block ~/.ssh, ~/.aws, ~/.config, .env files, and other credential stores
  • Temporary directories for output — If the server creates files, write to a temporary directory that gets cleaned up
const BLOCKED_PATHS = [
  ".ssh", ".aws", ".config", ".env",
  "credentials", "secrets", ".git/config"
];

function isPathBlocked(filePath: string): boolean {
  const normalized = filePath.toLowerCase();
  return BLOCKED_PATHS.some(blocked =>
    normalized.includes(blocked)
  );
}

// Check before any file operation
if (isPathBlocked(requestedPath)) {
  return { error: "Access to this path is not permitted" };
}

Network Restrictions

Control what network requests your server can make:

  • Allowlisted domains — If your server only needs to reach api.github.com, block all other outbound requests
  • No internal network access — Block requests to 127.0.0.1, localhost, 169.254.x.x (AWS metadata), 10.x.x.x, 192.168.x.x, and other private ranges
  • Protocol restrictions — Only allow HTTPS. Block file://, ftp://, gopher://, and other protocols

Resource Limits

Prevent denial-of-service, whether accidental or intentional:

  • Memory limits — Set --max-old-space-size for Node.js servers
  • Timeout limits — Every tool call should have a maximum execution time
  • Rate limiting — Limit how many times a tool can be called per minute
  • Output size limits — Cap the response size to prevent memory exhaustion

Common Vulnerabilities in MCP Servers

These are the vulnerabilities we see most often in MCP server code. Check your servers for each one.

1. Command Injection

The most dangerous vulnerability. It happens when user input is passed to a shell command without sanitization.

// VULNERABLE: input.filename could be "; rm -rf /"
exec(`cat ${input.filename}`, callback);

// SAFE: Use execFile with argument array
execFile("cat", [input.filename], callback);

// SAFER: Don't use shell commands at all
const content = await fs.readFile(input.filename, "utf-8");

2. Path Traversal

When a tool accepts a file path and the attacker uses ../ sequences to escape the intended directory.

// VULNERABLE: user sends path "../../../../etc/shadow"
const content = fs.readFileSync(path.join(baseDir, input.path));

// SAFE: Resolve and verify
const resolved = path.resolve(baseDir, input.path);
if (!resolved.startsWith(path.resolve(baseDir))) {
  throw new Error("Path traversal detected");
}
const content = fs.readFileSync(resolved);

3. Server-Side Request Forgery (SSRF)

When a tool that makes HTTP requests is tricked into requesting internal resources.

// VULNERABLE: user sends url "http://169.254.169.254/latest/meta-data/"
// This accesses AWS instance metadata and leaks credentials
const response = await fetch(input.url);

// SAFE: Validate URL before requesting
function isAllowedUrl(urlString: string): boolean {
  const url = new URL(urlString);

  // Block private IPs
  const hostname = url.hostname;
  if (hostname === "localhost" ||
      hostname === "127.0.0.1" ||
      hostname.startsWith("10.") ||
      hostname.startsWith("192.168.") ||
      hostname.startsWith("169.254.") ||
      hostname === "0.0.0.0") {
    return false;
  }

  // Only allow HTTPS
  if (url.protocol !== "https:") {
    return false;
  }

  return true;
}

4. Unvalidated Redirects

If your tool follows HTTP redirects, an attacker can redirect from an allowed URL to an internal resource. Either disable redirects or re-validate the URL after each redirect.

5. Information Leakage

Error messages that expose system internals — file paths, stack traces, database connection strings — help attackers understand your environment. Return generic error messages to the tool caller and log detailed errors internally.

try {
  const result = await riskyOperation(input);
  return { content: [{ type: "text", text: result }] };
} catch (error) {
  // Log the full error internally
  console.error("Tool error:", error);

  // Return a generic message to the caller
  return {
    content: [{ type: "text", text: "Operation failed. Check server logs." }],
    isError: true
  };
}

Security Checklist for MCP Developers

Use this checklist before publishing any MCP server:

  1. Input validation — Every tool parameter is validated with a schema library (Zod, Pydantic). No raw user input reaches the system.
  2. No shell interpolation — Zero uses of exec() or os.system() with string concatenation. All subprocess calls use argument arrays.
  3. Path safety — All file paths are resolved to absolute paths and verified against an allowed directory. Symlinks are handled.
  4. Network safety — Outbound requests validate URLs. Private IP ranges and non-HTTPS protocols are blocked.
  5. Secrets management — No hardcoded API keys or credentials. All secrets come from environment variables.
  6. Error handling — Errors are caught and logged internally. Generic messages are returned to callers. No stack traces leak.
  7. Resource limits — Timeouts, memory limits, and rate limits are configured. No single tool call can consume unlimited resources.
  8. Minimal permissions — The server requests only the access it needs. Read-only where possible. Narrow API scopes.
  9. Dependency audit — Dependencies are pinned and audited with npm audit or equivalent. No known vulnerabilities.
  10. Logging — All tool invocations are logged with timestamps, inputs (sanitized), and outcomes. Anomalies are detectable.

How to Audit Existing MCP Servers

If you are using MCP servers written by others — and most developers are — you should audit them before granting access to your system.

Step 1: Read the Source Code

Open source is only a security benefit if someone actually reads the code. Check for:

  • Uses of exec, spawn, eval, or Function()
  • File operations — what paths can it access?
  • Network requests — what URLs can it reach?
  • Dependencies — how many, and are they well-known?

Step 2: Check Permissions

What does the server's tool list look like? A JSON formatting tool should not have tools for file writing or network access. If the tool list is broader than the stated purpose, that is a red flag.

Step 3: Test with Malicious Input

Try path traversal, command injection, and SSRF payloads against each tool. If any of them succeed, do not use the server until the issues are fixed.

# Quick audit tests to run against any MCP server:

# Path traversal test
# Send path: "../../../../etc/passwd"
# Expected: Rejected by the server

# Command injection test
# Send filename: "test.txt; cat /etc/passwd"
# Expected: Treated as a literal filename, not executed

# SSRF test
# Send url: "http://169.254.169.254/latest/meta-data/"
# Expected: Blocked by URL validation

Step 4: Run with Minimal Privileges

Even after auditing, run third-party MCP servers with the least privileges possible. Use a separate user account, a container, or a VM. Do not run them as root or with access to your home directory.

Step 5: Monitor Behavior

After deployment, monitor what the server actually does. Check logs for unexpected file access, network requests to unusual destinations, or error patterns that suggest exploitation attempts. Automated monitoring tools can alert you to anomalies.

MCP security is not optional. As these servers become more common and more powerful, the consequences of poor security grow. The good news is that the fundamentals — validate input, limit access, monitor behavior — are the same principles that have protected software systems for decades. Apply them consistently and your MCP servers will be as secure as the rest of your stack.
