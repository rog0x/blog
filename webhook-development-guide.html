Webhook Development Guide

March 20, 2026 · 5 min read · WebhooksAPIBackend

Webhooks are how services talk to each other in real-time. Stripe sends a webhook when a payment succeeds. GitHub sends one when code is pushed. Getting them right means handling signatures, retries, and edge cases.

Signature Validation

Never trust a webhook without verifying its signature. The pattern is consistent across services:

// Stripe pattern
const sig = req.headers['stripe-signature'];
const event = stripe.webhooks.constructEvent(
  req.body, sig, webhookSecret
);

// GitHub pattern
const sig = req.headers['x-hub-signature-256'];
const hmac = crypto.createHmac('sha256', secret)
  .update(JSON.stringify(req.body))
  .digest('hex');
const valid = sig === `sha256=${hmac}`;

Retry Logic

Your endpoint will fail sometimes. Network issues, server restarts, rate limits. The sender will retry — usually with exponential backoff:

  • Attempt 1: immediate
  • Attempt 2: 1 minute
  • Attempt 3: 5 minutes
  • Attempt 4: 30 minutes
  • Attempt 5: 2 hours

Your handler must be idempotent — processing the same event twice should produce the same result. Use the event ID as a deduplication key.

Common Mistakes

  • Not responding with 200 quickly. Process async. Return 200 immediately, then handle the event in a background job.
  • Not validating signatures. Anyone can POST to your endpoint. Verify the HMAC.
  • Not handling duplicates. Retries mean the same event arrives multiple times.
  • Not logging failures. Dead letter queue for events that fail after all retries.
