DevOps with AI: Prompts for CI/CD, Docker, Kubernetes, and Monitoring

March 19, 2026 · 8 min read · DevOpsAIDocker

DevOps is configuration-heavy. CI/CD pipelines, Dockerfiles, Kubernetes manifests, monitoring rules, incident runbooks — all of it is repetitive enough that AI can generate 80% of it, and important enough that the last 20% needs your expertise.

Here's how to use AI prompts effectively for DevOps work.

CI/CD Pipelines

The prompt pattern that generates production-ready pipelines:

Create a GitHub Actions CI/CD pipeline for a [FRAMEWORK] app:

On push to feature branch:
  Lint → Test → Build → Deploy to preview

On merge to main:
  Full tests → Build → Deploy staging →
  Smoke tests → Manual approval → Deploy production

Include:
- Dependency caching
- Parallel test execution
- Slack notification on failure
- Rollback procedure

The key is specifying the workflow triggers AND the deployment targets. Without both, AI generates a half-pipeline that only builds but doesn't deploy.

Dockerfile Optimization

Optimize this Dockerfile for production:

[paste Dockerfile]

Check for:
1. Multi-stage build (separate build from runtime)
2. Layer caching (most-changing layers last)
3. Non-root user
4. .dockerignore exists
5. Minimal base image (alpine or distroless)
6. No secrets in the image
7. Health check instruction
8. Proper signal handling (SIGTERM)

Target: under 100MB final image for Node.js,
under 50MB for Go.

AI is excellent at Dockerfile optimization because the best practices are well-documented and consistent. It catches things humans miss — like forgetting to add a non-root user or copying node_modules into the final stage.

Kubernetes Deployments

Create Kubernetes manifests for [APP_NAME]:

- 3 replicas, rolling update
- Resource limits (CPU: 500m, Memory: 512Mi)
- Liveness and readiness probes
- HPA (scale on 70% CPU, min 2, max 10)
- ConfigMap for app config
- Secret for credentials
- Ingress with TLS
- Network policy (only allow from frontend namespace)

Output: YAML manifests + Helm chart.

Monitoring Setup

Set up monitoring for [APPLICATION]:

Metrics to track:
1. Request rate, error rate, duration (RED)
2. CPU, memory, disk usage
3. Database connection pool
4. Custom: [your business metrics]

Alerts:
- Error rate > 5% for 5 min → Critical
- p99 latency > 2s → Warning
- Disk > 80% → Warning
- Health check failing → Critical

Dashboard: Grafana with 4 panels.
Alert channel: Slack.
Tool: Prometheus + Grafana.

Incident Response

Create an incident response runbook for [SERVICE]:

Severity levels:
- P1: service down (15 min response)
- P2: degraded (30 min response)
- P3: minor issue (4 hour response)

For each severity:
1. Who gets paged
2. First diagnostic steps (what to check)
3. Rollback procedure
4. Communication template for users
5. Post-mortem template

Infrastructure as Code

Write Terraform for:

- VPC with public/private subnets
- ECS Fargate service for [APP]
- RDS PostgreSQL (Multi-AZ)
- ElastiCache Redis
- ALB with TLS
- CloudWatch alarms

Separate environments: dev, staging, prod.
Use modules for reusable components.
Include cost estimation.
